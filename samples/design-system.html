Best Practices

Guidelines for using the design system effectively.

Consistency

  • Always use design tokens instead of hardcoded values
  • Follow the spacing scale for margins, padding, and gaps
  • Use the color scale consistently across all components
  • Maintain consistent typography hierarchy
  • Apply the same border radius across similar elements

Accessibility First

  • Ensure sufficient color contrast (4.5:1 for text)
  • Provide visible focus indicators for all interactive elements
  • Use semantic HTML elements (button, nav, main, etc.)
  • Add appropriate ARIA labels when needed
  • Test with keyboard navigation and screen readers

Performance

  • Use CSS custom properties for theming (no JavaScript required)
  • Minimize animations for users with reduced motion preferences
  • Optimize images and assets for fast loading
  • Use CSS Grid and Flexbox for efficient layouts
  • Avoid unnecessary re-renders with proper component structure

Responsive Design

  • Design mobile-first, then enhance for larger screens
  • Use relative units (rem, em, %) instead of fixed pixels
  • Test on multiple devices and screen sizes
  • Ensure touch targets are at least 44x44px for mobile
  • Use responsive grids that adapt to container size

Implementation Guide

Step-by-step guide for implementing the design system in your project.

Step 1: Install the Design System

Copy the design-system.html file to your project and import it in your main HTML file.

<!-- In your HTML head -->
<link rel="stylesheet" href="design-system.html">

Step 2: Use CSS Custom Properties

Reference design tokens using CSS custom properties in your stylesheets.

/* Example button styles */
.button {
background: var(--ds-color-primary);
color: white;
padding: var(--ds-space-2) var(--ds-space-4);
border-radius: var(--ds-radius-md);
font-size: var(--ds-text-sm);
transition: all var(--ds-transition-fast) var(--ds-ease-default);
}

.button:hover {
background: var(--ds-color-primary-600);
}

Step 3: Customize the Theme

Override CSS custom properties to match your brand colors and typography.

/* Override in your CSS */
:root {
--ds-color-primary: #your-brand-color;
--ds-font-sans: 'Your Font', sans-serif;
}

Step 4: Build Components

Create reusable components using the design tokens.

<!-- Example card component -->
<div class="card">
<div class="card-header">
<h3 class="card-title">Title</h3>
</div>
<div class="card-body">
<p>Content</p>
</div>
</div>

Step 5: Test and Iterate

  • Test your implementation across different browsers and devices
  • Verify color contrast meets WCAG AA standards
  • Test keyboard navigation and screen reader compatibility
  • Gather feedback from users and iterate on the design
  • Document any custom components or patterns you create

Migration Guide

Guide for migrating from v1.0 to v2.0.

Color System Changes

The color system has been expanded to include 11-step scales for each color family. The base color variables remain the same, but new scale variables have been added.

/* v1.0 - Base colors only */
--ds-color-primary: #6366f1;

/* v2.0 - Base + scale */
--ds-color-primary: #6366f1;
--ds-color-primary-50: #eef2ff;
--ds-color-primary-100: #e0e7ff;
...
--ds-color-primary-950: #1e1b4b;

Backward Compatibility

All v1.0 CSS custom properties remain unchanged. The new scale variables are additive and do not break existing implementations.

Good news: No breaking changes. All existing code continues to work without modification.

FAQ

Frequently asked questions about the design system.

How do I customize the colors?

Click the "Customize" button in the header to open the customization panel. Use the color pickers to change any base color, and the entire scale will automatically regenerate. You can also export your custom configuration.

Can I use this design system in my project?

Yes! This design system is designed to be a canonical template that can be adapted for any project. Copy the HTML file and modify the CSS custom properties to match your brand.

Does it support dark mode?

Yes, dark mode is fully supported. Click the theme toggle button in the header to switch between light and dark modes. All colors automatically adjust for optimal contrast and readability.

Is it accessible?

Yes, the design system follows WCAG 2.1 AA guidelines. All color combinations meet minimum contrast ratios, all interactive elements are keyboard accessible, and focus indicators are visible.

How do I update font sizes?

Use the customization panel to adjust font sizes for each heading level. Changes are applied in real-time via CSS custom properties. You can also modify the CSS variables directly in your stylesheet.

Can I change the font family?

Yes, the customization panel includes a font family selector with options for Inter, System UI, Serif, and Mono. You can also specify a custom font stack.

Glossary

Key terms and definitions used in the design system.

Design Token
A named entity that stores a visual design attribute (color, spacing, typography, etc.). Tokens are the building blocks of a design system.
CSS Custom Property
A CSS variable (e.g., --ds-color-primary) that can be reused throughout a stylesheet. Enables dynamic theming and consistent design values.
Color Scale
A series of related colors from lightest (50) to darkest (950). Each step is optimized for specific use cases (backgrounds, text, borders).
Semantic Color
A color named by its purpose rather than appearance (e.g., "primary", "success", "error" instead of "blue", "green", "red").
Component
A reusable UI element built with design tokens (e.g., button, card, input). Components ensure consistency across the application.
Accessibility (a11y)
Designing and building interfaces that can be used by everyone, including people with disabilities. Includes keyboard navigation, screen reader support, and sufficient color contrast.
Responsive Design
An approach to web design that makes pages render well on a variety of devices and screen sizes. Uses flexible grids, layouts, and CSS media queries.
Dark Mode
A color scheme that uses light-colored text on a dark background. Reduces eye strain in low-light environments and can save battery on OLED screens.

Contributing

Guidelines for contributing to the design system.

Adding New Components

  • Ensure the component uses only design tokens (no hardcoded values)
  • Test the component in both light and dark modes
  • Verify accessibility (keyboard navigation, screen readers, contrast)
  • Add the component to the appropriate section in this documentation
  • Include usage examples and best practices

Modifying Existing Tokens

  • Changes to tokens affect all components using them
  • Test thoroughly before making breaking changes
  • Update the JSON-LD manifest to reflect changes
  • Document the change in the changelog
  • Consider backward compatibility when possible

Code Style

  • Use CSS custom properties for all design values
  • Follow BEM naming convention for component classes
  • Keep CSS specificity low and predictable
  • Use semantic HTML elements
  • Write clear, descriptive comments for complex logic

Final Notes

Additional notes and considerations for using this design system.

Customization Tips

  • Start with the base colors and let the scale generator create the full palette
  • Test your color choices in both light and dark modes
  • Use the export feature to save your custom configuration
  • Consider accessibility when choosing colors (contrast ratios matter)
  • Keep typography hierarchy clear and consistent
  • Use spacing tokens consistently throughout your application

Common Pitfalls

  • Don't hardcode color values — always use CSS custom properties
  • Don't skip accessibility testing — use contrast checkers
  • Don't ignore responsive design — test on multiple screen sizes
  • Don't forget dark mode — ensure colors work in both themes
  • Don't overuse animations — keep them subtle and purposeful
  • Don't mix design systems — stick to one consistent approach
